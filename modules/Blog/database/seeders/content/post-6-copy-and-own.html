Most software starter kits give you a locked box. You get the features, but you also get the constraints. Want to change how authentication works? Fork the package and maintain the fork yourself. Want to add a field to the billing model? Hope the package has an extension hook for that. Want to understand why something works the way it does? Read documentation, not source code.

Saucebase takes the opposite approach: copy-and-own.

What copy-and-own means

When you install a Saucebase module, the source code lands in your repository under modules/. It's tracked in version control. It's yours to read, edit, extend, and break. There's no vendor lock-in, no abstraction layer between you and the code that runs your business logic.

This isn't a new idea — it's how Laravel itself recommends building things. Publish vendor files, customise them, own them. Saucebase applies that philosophy at the module level.

The practical benefit

When a bug appears in your authentication flow, you look at modules/Auth/app/Http/Controllers/. When you want to add a field to the billing portal, you edit modules/Billing/resources/js/pages/. There's no mental overhead of figuring out which hook to use or which config key controls the behaviour you need. You find the code and change it.

The tradeoff

When Saucebase ships improvements to a module, you don't get them automatically. But this is the right tradeoff. A module that runs your business should be something you understand and control — not something you blindly upgrade and hope nothing breaks.

Building your own modules

The recipe system makes it trivial to scaffold new modules that follow the same conventions:

php artisan saucebase:recipe ModuleName

Your custom features get the same structure, the same patterns, the same copy-and-own philosophy. They feel like first-class citizens because they are.

This is the sauce. These are your recipes. Build on them.
